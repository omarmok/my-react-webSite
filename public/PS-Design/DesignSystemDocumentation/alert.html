Alerts

Alerts communicate important states and guide users toward follow-up actions. Use them inside flows to highlight critical, warning, info, or success states that need attention.

نجاح: تمت العملية بنجاح ويمكن المتابعة.
المكوّن: التنبيهات الإصدار: قيد التحديث آخر تحديث: 10 مايو 2026

Purpose & when to use

Alerts are system messages used to convey important information or states requiring user attention. Various types of alerts (info, success, warning, error) are available, each designed for a specific use case.

  • Surface alerts within flows to highlight outcomes that need follow-up (success, error, warning, info).
  • Do not use alerts decoratively; pair a clear message with an icon and concise copy.
  • Attach an explicit action (e.g., “View details”) so users can resolve the situation.
  • Describe the state in text when hiding color or icons to keep the intent accessible.

Anatomy

The .moi-notification component uses a BEM structure. Visual state is driven by data-status, not CSS modifier classes.

Element Class Description
Root .moi-notification Root container. Takes data-status for visual state. Add alert alert-dismissible fade show only when Bootstrap dismiss behavior is needed.
Content .moi-notification__content Wraps the icon, title, and message.
Icon .moi-notification__icon Icon wrapper. Inner SVG must use currentColor. Add aria-hidden="true" on the wrapper.
Title .moi-notification__title Primary alert headline, typically a <strong>.
Message .moi-notification__message Supporting message text.
Actions .moi-notification__actions Wraps the action button and close button.
Action button .moi-notification__btn Action button. Used with btn. Button variant is applied automatically by JavaScript based on the parent's data-status.
Close button .moi-notification__close Dismiss button. Add data-bs-dismiss="alert" for Bootstrap dismiss. Must have a clear aria-label.

data-status system

  • Use data-status instead of CSS modifier classes. Supported values: neutral, info, error, warning, success.
  • Do not add status classes manually — data-status handles it via CSS and JavaScript.

SVG & currentColor

  • Inline SVG must use currentColor to inherit the status color automatically.
  • Do not hardcode color fill values inside the component's SVGs.

Border rule

  • Notification uses a bottom border only: border-bottom: 3px solid status-color.
  • Do not add borders to other sides of the notification.

Button behavior

  • Always use btn moi-notification__btn. The correct variant class is added by JavaScript based on data-status.
  • Do not manually specify the button variant in HTML.

ID rule

  • Do not rely on IDs for styling or behavior. Use classes and data-status.
  • Avoid duplicate IDs on the same page.

ARIA & behavior

  • Set role="alert" for critical states.
  • Add aria-hidden="true" on the icon wrapper.
  • Close button must have a clear aria-label.

Alert Variants

The type of alert (and its color) is determined by the data-status attribute on .moi-notification. Do not add status modifier CSS classes directly.

Status data-status value Button variant (JS) Usage
Neutral data-status="neutral" moi-btn--black For general alerts or information with no positive or negative connotation.
Info data-status="info" moi-btn--Info For non-critical information or guidance.
Error data-status="error" moi-btn--Error To indicate a serious problem or failure in an operation.
Warning data-status="warning" moi-btn--Warning To warn about potential issues or actions requiring attention.
Success data-status="success" moi-btn--Primary To confirm an action completed successfully.

Usage rules

  • Use .moi-notification with data-status to set the alert state. Do not add status classes manually.
  • Add alert alert-dismissible fade show only when Bootstrap dismiss behavior is required.
  • The component uses a bottom border only: border-bottom: 3px solid status-color.
  • Do not rely on IDs for styling or behavior. Use classes and data-status.
  • Test icons and buttons via keyboard and keep focus outlines intact.

Implementation Notes

Specific rules to follow when implementing the .moi-notification component.

Bootstrap integration

The following classes are optional and used only for Bootstrap dismiss behavior:

alert alert-dismissible fade show
  • Add data-bs-dismiss="alert" on the close button.
  • Do not use these classes for styling — they are for JavaScript behavior only.

Border rule

The notification uses only a bottom border, 3px thick, in the status color.

border-bottom: 3px solid <status-color>
  • Do not add borders on other sides.

Action button behavior

The button always stays btn moi-notification__btn. JavaScript adds the correct variant based on data-status.

const map = {
  neutral: "moi-btn--black",
  info:    "moi-btn--Info",
  error:   "moi-btn--Error",
  warning: "moi-btn--Warning",
  success: "moi-btn--Primary",
};

SVG rule

All SVGs inside the component must use currentColor to inherit the status color. fill="white" is only acceptable for the background shape in composite icons (e.g., warning, success).

<path fill="currentColor" d="..." />

Accessibility Guidance

  • Use role="alert" for critical/high-impact alerts (e.g., errors). For less urgent states, consider role="status" with aria-live="polite".
  • Icons within alerts should be decorative (aria-hidden="true") unless they convey additional meaning not present in the text.
  • Ensure all interactive buttons (e.g., 'View Details' or dismiss button) are keyboard-focusable and have clear labels (aria-label).
  • Avoid relying on color alone to convey meaning. There should always be supplementary text or an icon to communicate the alert's state.

Edge cases

  • Do not stack multiple alerts for the same state in a row; give each alert its own spacing.
  • For long text, keep it within .font-md-400 and break in a new line to avoid clipping.
  • When multiple states exist (error + warning), show separate alerts with clear labels.
  • If an action needs confirmation, attach a dialog link rather than repeating the alert.
