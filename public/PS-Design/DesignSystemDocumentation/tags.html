Purpose & when to use

use short labels to sort items or flag workflow status, and choose emphasis based on importance.

Descriptors & metadata

  • Non-interactive tags that mark category or source.
  • Use subtle styles like .defualtprioroty.
  • Keep text to one or two words to avoid wrapping.

Status

  • Map to meaning: .highpriority for errors or delays.
  • Use filled variants when higher visual prominence is needed.
  • Add role="status" for non-interactive updates.

Filtering & interaction

  • Use a <button> when the tag is clickable or removable.
  • Include a close icon or text for removable filters.
  • Keep at least 8px spacing between adjacent tags.

Usage rules

  • Keep tag text short (two words max) to avoid wrapping.
  • Apply role="button" to interactive pills and include aria-label on remove icons.
  • Ensure close/remove icons meet minimum touch size (~40px).
  • Use tags for categorization, filtering, or status—not for purely decorative content.

Anatomy

Element Description
Container .moi-tag provides the pill shape and padding.
Label Text uses .font-md-500 for readability.
Icon Icons live inside .icon and inherit semantic color classes like .Primary-600.
Action Interactive tags use a <button> with a clear aria-label or a remove icon.

Hover / focus

  • Hover elevates the pill slightly while keeping the color consistent.
  • Focus-visible reveals a clear outline for interactive tags.

Active / selected

  • Selected tags may add a bolder background or check icon via accented classes.
  • Use .is-active or a modifier class to highlight the active state without structural changes.

Disabled / non-interactive

  • Add .is-disabled to gray out the tag and block interactions while keeping it in flow.
  • Non-interactive tags do not need aria-labels but keep a textual cue nearby.

Do & Don't

Do

  • Keep labels short and clear like "High" or "In review".
  • Reuse the same status color set across the product to build familiarity.
  • Reach for filled tags only when stronger emphasis is necessary.

Don't

  • Use tags as primary calls to action.
  • Mix more than three color families in one view.
  • Rely on color alone; the label must explain the meaning.

DesignOps Governance – Tag Usage

These rules are mandatory to keep tag usage consistent and auditable across all products and features. Any exception requires prior DesignOps approval with documented rationale and impact.

Inquiry / View

On inquiry, detail, review, and audit screens, a tag is a read-only status indicator. It must read as passive information, not as an editable or selectable control.

  • Use for display only: Show only the current state or descriptor inside the tag on detail, summary, and review screens.
  • Do not imply editing: Avoid edit affordances, remove icons, dropdown carets, or pointer cues unless a real action exists.
  • Keep extra detail outside the tag: If the screen needs explanation or business detail, place it in a nearby details or requirements block.
  • Support audit flows: The label must be self-explanatory and stable across screens so reviewers can scan and verify it quickly.

Read-only example

Interaction

Tags become interactive only when they perform a real action such as opening a view, applying a filter, or removing a selection. In those cases they must be implemented as proper, accessible controls.

Pattern Use when Expected behavior
Non-clickable tag When the tag is only a displayed status or descriptor in a view screen, table, or card. Use a <span>, with no pointer cursor, click action, remove action, or link behavior.
Clickable tag When the tag applies a filter, opens a related result, or drills into a linked view. Use a <button> or <a> depending on behavior, provide clear hover/focus feedback, and support Enter and Space for buttons.
Closable tag When the tag represents a filter or selection that the user can explicitly remove. Keep only one action inside the tag, label the close control with aria-label, and make the target keyboard reachable.

Edge cases

Handle the following cases consistently so tags do not break visually or behaviorally in dense layouts or incomplete data scenarios.

  • Unknown status: Never expose an undocumented raw value to users. Map it to an approved neutral state or resolve the documentation gap before release.
  • Missing status label: Do not render an empty shell. Provide an approved user-facing label before render, or hide the tag until data is complete.
  • Long label text: Shorten the label first. If that is not possible, allow wrapping within the container without breaking layout or creating horizontal overflow.
  • Icon + label: The icon supports meaning but does not replace text. Keep a visible label and ensure the icon inherits currentColor.
  • Closable tag: Keep the remove button as the only action inside the tag, give it a clear accessibility label, and do not rely on color alone to explain removal.
  • RTL text: Preserve natural text and icon order in Arabic and English, and verify alignment and reading direction after switching languages.
  • Small containers: Use a smaller size, allow wrapping, or move the group into a scroller/collapsible area. Never let tags spill outside the container.

Labeling & Wording Rules

Tag wording must stay business-friendly, short, and easy to read in Arabic and English. The data-status value is for developers, while the visible label is for users.

  • Keep labels short: Prefer 1–3 words that describe a state or attribute, not an instruction sentence or long explanation.
  • Do not show technical codes: Never expose values such as underprogress or highpriority to end users.
  • Keep Arabic readable: Use clear Arabic terminology that scans well, and avoid obscure abbreviations or awkward transliterations.
  • Use consistent naming: One meaning should have one approved label across products and screens, with no competing synonyms.
  • Separate the layers: The visible user label may differ from the implementation value, but both must be documented together and mapped clearly.

  • On add/create and manage screens: Show only the essential tags, ideally no more than four per row, to avoid visual noise.
  • Supporting requirements: Place business rules, attachments, and inputs below the tag group with a clear title such as “Business Requirements” or “Feature Details.”
  • Do not use tags as inputs: If the user needs to choose, edit, or add a value, use an appropriate form control instead of the tag itself.

Rationale (Why This Rule Exists)

These principles explain why the tag stays simple, readable, and auditable across design, engineering, and content teams.

UX

Keeps the tag a quick scannable surface instead of overloading users with unexpected information or actions.

DesignOps

Prevents uncontrolled naming drift and makes usage easier to audit and standardize across products.

Accessibility

Separates informational tags from interactive controls, which clarifies expectations for keyboard and screen-reader users.

Consistency

Ensures status values, colors, and labels behave the same way in tables, cards, and detail views.

Developer handoff

Separates visible copy, implementation values, and business rules, which reduces ambiguity during implementation and maintenance.

Developer API

This section shows the pattern and attributes for both interactive tags and passive pills. 

<!-- Interactive tag with icon and close button -->
<span class="moi-tag" data-size="md" data-status="underprogress">
  <span class="moi-label">In review</span>
  <button class="tag-close" type="button" aria-label="Remove tag">
    <span class="icon icon-md"><svg viewBox="0 0 24 24" aria-hidden="true">...</svg></span>
  </button>
</span>
<!-- Passive pill -->
<span class="moi-tag" data-size="md" data-status="defualtprioroty"><span class="moi-label">Neutral</span></span>
  • Each tag uses .moi-tag with data-status and only the optional data attributes it needs.
  • The remove button needs an aria-label and focusable target for keyboard users.
  • Non-interactive tags stay as span with no special roles.
  • Keep icons visible; mark them aria-hidden="true" and provide text if needed.

Behavior notes

  • data-status controls text color, background, border, and dot color.
  • Dot is visible by default.
  • Dot is hidden only when a leading direct svg/img exists.
  • Close button does not hide the dot.
  • Icons inherit currentColor.
  • Icon size follows tag size.
  • Typography remains inherited.

Accessibility

  • Apply role="status" to non-interactive status tags.
  • Use a <button> with visible focus styles for clickable tags.
  • Ensure 4.5:1 contrast on text, especially on filled variants.
  • Announce state changes (e.g., remove/add) with an aria-live region.
