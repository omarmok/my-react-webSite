Card

Card surfaces encapsulate decisions, inputs, and actions inside a consistent container with official spacing tokens.

Component: Card Version: TBD Last updated: January 28, 2026

Overview

Cards surface related data, controls, and context in a single container that inherits our spacing, radius, and elevation tokens.

What is the Card?

  • The root element is .moi-card; prefer data-status="create|edit|details" and use .moi-card--legacy-create|edit|details only for compatibility, while keeping padding and elevation consistent.
  • Header content lives in .moi-card__header with the .moi-card__title-group block that pairs an icon with a title line and optional supporting text.
  • The body is .moi-card__body and accepts forms, read-only details, or mixed content while keeping consistent gutter spacing.

When to use

  • Group a form and its actions inside a single logical surface.
  • Bound modal-like sections that need a tertiary background.
  • Surface validation, guidance, or state updates along with the controls.

When not to use

  • Avoid stacking cards without a clear section context.
  • Do not overload the header with multiple competing actions.
  • Choose layout grid rules instead of a card when the layout requires column alignment beyond 1 card width.

Anatomy

Header

  • The header sits on .moi-card__header and keeps top-radius and border tokens in sync.
  • Use .moi-card__title-group to wrap an icon, title text, and optional subtitle.
  • Only one contextual header action is allowed so the tab order stays predictable. Extra buttons should move to the body or footer.

Body

The .moi-card__body supplies the inner padding and should host form fields, read-only labels, or a mix that still looks cohesive.

Footer (optional)

Footers appear when there is a dominant action row—use .moi-card__footer to align buttons and keep spacing consistent.

Card Container

Root element .moi-card unifies padding, boundaries, and elevation across all variations.

Header

Header sits on .moi-card__header and pairs an icon + title block with a single contextual action to keep focus order simple.

Body

Body uses .moi-card__body, acting as a flexible container that can host any content (forms, details, alerts, CTAs) while maintaining spacing tokens.

Footer

Footer (optional) uses .moi-card__footer to align primary/secondary buttons and complete the action strip.

Content Guidance

Card bodies flex to host forms, read-only details, tables, alerts, lists, and CTA rows while preserving the spacing, typography, and hierarchy tokens that keep everything aligned.

  • Wrap forms, paired read-only labels, tables, alerts, lists, and CTAs inside the body so the spacing system keeps everything organized.
  • Rely on official padding, gap, and typographic tokens and avoid hard-coded measurements inside the shell.
  • Group primary and secondary actions in the footer or a dedicated action strip so button hierarchy and tab order stay predictable.
  • Place cards inside clearly labeled sections to keep the context consistent across enterprise screens.

Do & Don’t

✔

Do

  • Use official spacing, radius, and elevation tokens instead of hard-coded values.
  • Keep the header action singular so focus order remains predictable.
  • Group buttons inside the body or footer with clear primary/secondary hierarchy.
✕

Don’t

  • Avoid inline CSS or custom width overrides on card sections.
  • Do not mix read-only details with interactive fields inside the same card body.

DesignOps Governance

The governance framework keeps the Card predictable across enterprise surfaces and reinforces token-led decisions.

Purpose / Intent

Cards act as a trusted decision surface that aligns formal actions with the system tokens.

Content rule: Flexible body

The Card body can host forms, details pairs, tables, alerts, lists, CTA rows, and mixed contextual sections while honoring the spacing, typography, and hierarchy tokens.

  • Forms, read-only pairs, tables, alerts, lists, and CTA rows are all supported inside the body.
  • Keep padding, gaps, typography, and hierarchy tied to the official tokens—do not hardcode new measurements or overrides.
  • Maintain vertical rhythm and responsive wrap rules (for example, details grids rely on .w-25 wrappers) so cards adapt without overflow.

Standardization

  • Header structure should pair an icon with the title and only expose one contextual action.
  • Rely on the official color, spacing, and typography tokens for every card surface.
  • Avoid bespoke styling or overrides inside the Card shell.

QA / Acceptance Criteria

  • Inner padding and gaps follow the approved token system.
  • Header alignment stays consistent across LTR and RTL layouts.
  • Primary and secondary buttons live in the footer or action strip with clear hierarchy.
  • Body content must not overflow and should wrap responsively at smaller breakpoints.
  • Background and text contrast comply with the enterprise palette.
  • Focus order and keyboard navigation remain predictable.

Maintenance

  • When a new card variation is added, update the documentation, capture a screenshot/example, and refresh the changelog (if one exists).
  • Keep naming aligned with the component naming conventions.

Code Examples

Copy-paste snippets for Create, Edit, and Details cards using either the preferred data-status API or the legacy modifier fallback.

Preferred usage

  • Use <div class="moi-card" data-status="create"> and switch the status value to edit or details as needed.

Legacy compatibility

  • Use <div class="moi-card moi-card--legacy-create"> when data-status is unavailable; swap to moi-card--legacy-edit or moi-card--legacy-details for the other states.
  • The header action button must use an official current button modifier such as .moi-btn--Neutral, .moi-btn--Info, or .moi-btn--Primary depending on the card state.
