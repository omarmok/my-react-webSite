File Uploader

A document-upload component pairing a required descriptive name with a drag-and-drop zone, live progress, a file list, and a type-aware preview modal — built entirely on PS-Design tokens.

Component: File Uploader Version: 1.2 (Accessibility & Preview enhancements) Last updated: June 16, 2026

Overview

.moi-file-uploader is the official attachment-upload component for government systems. It binds every file to a descriptive name the user must enter before the dropzone unlocks, so every document that enters the system already has a meaningful name from the moment it arrives — not just whatever filename the user's device happened to assign it.

When to use it

  • Requesting one or more official attachments that must be clearly named (ID scan, contract, report, evidence file).
  • The reviewer needs to preview the file in-page, without leaving the screen or opening a separate application.
  • You need clear client-side limits on accepted file types/size before anything reaches the server.

When not to use it

  • A single simple avatar/profile-picture upload — use a lighter, dedicated image-upload control instead.
  • Flows that never need a descriptive name — this component always requires one by design.
  • Routine uploads of very large files — the built-in limit is 5 MB per file.

Typical use cases

  • Government portals — forms that require official attachments with a clear descriptive name.
  • Citizen services — submitting ID documents or supporting paperwork for a service request.
  • Internal employee systems — uploading reports, forms, or administrative attachments.
  • Case management systems — attaching documents to a specific case with a name that stays traceable later.
  • Evidence/document submission — where naming accuracy and predictable behavior matter most.

Component Anatomy

The component is two independent cards (name field/dropzone, and the uploaded-files list) plus a preview modal, all built in BEM under the root .moi-file-uploader class.

Title & Description

The page heading and intro line (outside the component itself) explain the purpose of the upload before the user starts.

Name Field

A required text input (.moi-file-uploader__input) holding the descriptive name. The dropzone stays disabled until it's filled.

Validation State

.is-err / .is-ok modifier classes on the input, paired with a live error hint and a synced aria-invalid attribute.

Success State

A checkmark inside the input plus a "اسم محفوظ" hint. Both are automatically cleared once the first upload completes, via resetNameField().

Dropzone

A click + drag-and-drop target (role="button" tabindex="0"), visually and semantically disabled (aria-disabled) until the name is valid.

Upload Progress

A temporary row per in-flight file with a real role="progressbar" and live aria-valuenow, plus an immediate cancel action.

Uploaded Files List

An independent card (#filesSection) that stays fully hidden until the first upload succeeds, and hides again if every file is deleted.

File Actions

A preview and a delete button per row. Delete uses a two-tap "armed" confirmation (.armed) instead of a separate confirmation dialog.

Preview Modal

A large role="dialog" aria-modal with a full focus trap, rendering the best possible preview for the file's detected type.

Download Action

A footer button that rebuilds the original file from the stored File object and downloads it under its original filename.

Delete Action

A footer button using the same two-tap confirm pattern, which deletes the file and closes the modal in one step.

File Preview Support

getPreviewKind(f) decides the preview type by checking the file's real MIME type (f.file.type) first, then falling back to the extension when the MIME type is missing or unreliable — exactly as implemented in the final component.

File Type Preview Support Behavior
JPG / JPEG / PNG Inline preview Rendered as a large <img> inside .preview-image-wrap (responsive height up to 680px), using the object URL created at upload time and stored as f.dataUrl.
PDF Inline preview An <object type="application/pdf"> with a fresh object URL (#view=FitH) in a large area (.preview-pdf-wrap, up to 680px); created and revoked on every modal open/close.
TXT / CSV / JSON / XML / HTML / CSS / JS / SCSS / MD Safe text preview Read via FileReader.readAsText; content is injected into <pre><code> using textContent (never innerHTML), which guarantees any HTML/script inside the file is auto-escaped, not executed. Truncated at 50,000 characters; code is forced to LTR direction regardless of the page's RTL context.
DOCX / XLSX / PPTX / DOC / XLS / PPT Attempt + fallback The fallback panel is wrapped inside <object data="blob:..." type="<official mime>">. If the browser can't render that type inline (true for virtually all mainstream browsers today), the browser's own native fallback mechanism kicks in and shows the panel — clear metadata + the required message + the footer's download action.
Other / unsupported files Metadata fallback A metadata panel (descriptive name, original name, type, size) with the same unsupported-type message and the footer download button — no file type ever produces an empty or broken modal.

Unified fallback message

Both the Office-file branch and the truly-unsupported branch render the exact same Arabic copy: "لا يمكن عرض هذا النوع داخل المتصفح مباشرة، يمكنك تحميل الملف أو فتحه من جهازك." Do not invent a different message for this case in other pages or teams.

Accessibility

This section documents every accessibility decision that is actually implemented in the final code — not a wishlist.

  • RTL support — the hosting page declares <html lang="ar" dir="rtl">, and every positioning rule in the SCSS uses logical properties (inset-inline-start/end, padding-inline-start/end) instead of left/right, so it mirrors correctly with zero RTL-specific overrides.
  • Keyboard navigation — every interactive element (the field, the dropzone, the preview/delete/cancel/download buttons, the footer buttons) is reachable by Tab in a logical order that follows DOM order.
  • Dropzone keyboard support — the element carries role="button" tabindex="0" and responds to Enter and Space via onZoneKeydown(). While disabled (aria-disabled="true") it stays focusable, and both mouse and keyboard activation redirect the user to the name field identically — there's no behavioral gap between input methods.
  • Focus management — opening the preview stores the triggering element (lastTrigger) and immediately focuses the dialog box itself (tabindex="-1"); closing it returns focus to that exact element.
  • Modal focus trap — a single global keydown listener computes the focusable elements inside #modalBox only while it's open, and redirects Tab/Shift+Tab between the first and last one — focus can never escape to the page content behind the modal.
  • Escape close — the same global listener closes the modal immediately on Escape, regardless of which element inside it currently has focus.
  • Screen reader support — a visually-hidden (sr-only) #uploaderStatus region with role="status" aria-live="polite" announces each file's start / success / cancel / delete — announcements that are never meant to be seen, only heard.
  • Required field semantics — the input carries required and aria-required="true", with aria-invalid kept live-synced to the .is-err/.is-ok state instead of relying on color alone.
  • Upload progress accessibility — every uploading row has a real role="progressbar" with live aria-valuemin/max/now, plus an aria-label carrying the filename — the screen-reader experience matches what a sighted user sees, not a silent gap.
  • ARIA labels — every icon-only button (preview, delete, cancel, close) carries an explicit text aria-label rather than relying on title alone, and every purely decorative icon carries aria-hidden="true".
  • WCAG 2.1 AA considerations — secondary text color was raised from Neutral-400 to Neutral-500 (2.51:1 → 4.78:1) to clear the 4.5:1 text threshold, and the same change was applied to action-button icons to clear the 3:1 non-text threshold; every other badge/icon color pair was checked and sits above 7:1.
Implementation guidance If you clone this component onto another page, copy the entire JavaScript block with it — the focus-trap logic, the aria-disabled syncing, and resetNameField() are interdependent, and copying only the markup without its script silently breaks accessibility.

Content Guidelines

File Names

  • Ask for a name that describes the file's content, not just its type or source — "صورة الهوية الوطنية" ("National ID photo"), not "صورة" ("Image") or "مستند1" ("Document1").
  • Use a placeholder showing a real example ("مثال: صورة الهوية الوطنية") instead of generic instructional text like "اكتب هنا" ("Type here").
  • When the same descriptive name repeats, it is auto-numbered ("name (2)") with a brief tip explaining the auto-rename happened — never make this change silently.
  • The original filename is always preserved and shown alongside the descriptive name, so no technical context is lost for a later reviewer.

Error Messages

  • State the violated constraint and the consequence together in one short sentence: "ملف تجاوز الحد الأقصى (5 ميجابايت) — لم يتم رفعه" ("File exceeds the maximum (5 MB) — it was not uploaded").
  • When a file type is rejected, name the types that actually are allowed — never just "نوع غير مدعوم" ("Unsupported type") with no indication of what would work instead.
  • Required-field messages must name the action needed: "أدخل اسمًا قبل رفع الملف" ("Enter a name before uploading"), not "حقل غير صالح" ("Invalid field").
  • Every error must reach screen-reader users via a real role="alert", never through a color change alone.

Upload Instructions

  • Lead with the verb ("اسحب", "اختر" — "drag", "choose"), then state the constraint (allowed types, max size) on a visually lighter second line.
  • Keep the constraint text (types/size) permanently visible in the dropzone, not only after a rejection — users should know the limit before they try, not after they fail.
  • When the dropzone is disabled for lack of a name, explicitly redirect the user to the name field rather than leaving it disabled with no explanation.

Success Messages

  • Keep success confirmation calm and non-blocking (a checkmark + a short phrase), not a popup dialog for every low-stakes upload.
  • After a successful upload, reset the field to its initial state automatically (clear the name, hide the success state, re-disable the dropzone) so the user is ready to type a fresh name for the next file instead of accidentally reusing the previous one.
  • Announce success to screen-reader users too, through the live region — not only via a visual change in the list.

Design System Governance

Purpose

This component exists to remove the need to rebuild upload logic (descriptive naming, validation, progress, multi-type preview, safe deletion) from scratch in every government system. A team that reimplements these requirements on its own re-introduces the accessibility gaps already solved here once.

Standardization

  • Use .moi-file-uploader as-is without modifying the root markup — add new file types only through the TYPE/ALLOWED tables.
  • Never clone a markup fragment without its script — state (savedName, blurred, uploadJobs) lives entirely in JavaScript.
  • Colors, spacing, and radii are all PS-Design tokens (Neutral/Info/Primary/Warning/Lavender/Error) — no new hard-coded hex values.

Reuse Rules

Reuse the component as-is for any single- or multi-attachment flow that needs a descriptive name. Create a new variant only when a core business rule actually differs — e.g. no descriptive name at all, or uploads with no type/size validation. Never fork the file just to change copy or a color; those are passed-in values, not reasons to diverge.

QA / Acceptance Criteria

  • Upload works — the progress bar starts, completes, and the file moves into the done list automatically.
  • Validation works — the dropzone cannot open without a valid descriptive name, and error messages show and hide correctly.
  • Accessibility works — full keyboard support, the focus trap holds, Escape closes, and a screen reader hears every live announcement.
  • Preview works — every supported category (images, PDF, text, Office, unsupported) renders useful content with no empty or broken modal.
  • Download works — the footer download button restores the original file with its correct extension and name, for every type.
  • RTL works — no mirrored-incorrectly visuals, and the icon never overlaps the input text.
  • Mobile works — the modal footer wraps to full-width buttons, and preview areas shrink responsively across the three breakpoints.

Maintenance

  • When adding a new file type, update TYPE/ALLOWED/OFFICE_MIME together, and update the dropzone subtitle and error copy so they stay honest about the actual allowed list.
  • When changing a color token to fix contrast, re-check the actual ratio (never eyeball it) and record the old/new value in the changelog.
  • Update this page whenever the component's behavior changes — it documents the final, currently-shipped version, not the original one.

Do & Don't

✔

Do

  • Use a real descriptive name: "عقد التوريد النهائي 2026" ("Final 2026 supply contract").
  • Leave the dropzone disabled until the name is valid — that's intended behavior, not a bug.
  • Add a clear aria-label to any new icon-only button you add to a file row.
  • Copy the markup and the script together when reusing the component on another page.
✕

Don't

  • Don't use generic names like "ملف" ("File"), "مستند1" ("Document1"), or just reusing the raw original filename.
  • Don't delete a file on a single click — use the existing two-tap (.armed) confirmation pattern.
  • Don't rely on color alone to show error or success — always keep the text and aria-invalid in sync.
  • Don't show the files card before the first successful upload — that breaks the progressive-disclosure behavior intentionally built into the final version.

Correct upload behavior

A file that's validated for size and type, starts progress immediately, moves itself into the done list, and clears the name field afterward — no manual cleanup step needed.

Incorrect upload behavior

Uploading a file past the size/type limit, or allowing it through without a valid descriptive name — both must be rejected before the file ever reaches the server, not after.

Accessible usage

Opening the dropzone with Enter/Space, tabbing all the way to the download button, and hearing "تم رفع الملف بنجاح" from a screen reader without looking at the screen at all.

Inaccessible usage

Adding a new icon button with no aria-label, removing the hidden attribute from the files list without updating screen-reader focus, or relying on title alone as the button's accessible name.

Best Practices

  • Never change MAX_SIZE or ALLOWED without also updating the dropzone subtitle and the type-error copy — all three must always stay in sync.
  • When adding a new type to the Office list, also add its correct MIME in OFFICE_MIME — otherwise it falls back to application/octet-stream, which stops the browser from even attempting a real render.
  • Never swap textContent for innerHTML in the text-file preview for any reason — that's precisely what keeps it safe from injection.
  • Any new button you add inside the preview modal is automatically included in the focus trap, because getFocusable() queries focusable elements dynamically — you never need to touch the trap logic itself.
  • When assigning a color to a new file type, pick a token family not already used in this component (e.g. Gold or Brown) instead of reusing Error, which is reserved for actual error states.
  • Recompile main.css (via sass) and verify before assuming a change works — several rules here depend on cascade order (e.g. [hidden] needing to out-specificity .moi-card's display: flex).
