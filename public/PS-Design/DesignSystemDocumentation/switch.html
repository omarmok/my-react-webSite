Switches

Switches translate binary settings into a single tap. Pair the Bootstrap .form-check-input with .form-check.form-switch and a descriptive label so that both mouse and keyboard users see a clear relationship.

Overview

The switch is a checkbox with role="switch", rendered by .form-check.form-switch plus .form-check-input. The SCSS at css/common/_Switch.scss layers the track, thumb, and motion rules.

When to use / When not to use

When to use

  • Toggle a single boolean setting or feature flag with immediate feedback.
  • Surface an on/off choice inline within a settings list, not as a separate panel.
  • Pair the switch with a concise label that describes the destination state.

When not to use

  • Avoid using switches for navigation or action triggering.
  • Do not force multi-state decisions into a binary switch.
  • Do not delay feedback with asynchronous saves or secondary confirmations.

Anatomy

Structure is derived from the selectors in the SCSS.

  • Root: .form-check.form-switch provides layout and spacing.
  • Track: .form-check-input defines the 48×24 track, border, and background.
  • Thumb: .form-check-input::before renders the 16px thumb.
  • Label: .form-check-label sits next to the track with a 10px gap.
  • States: :checked, :disabled, and :focus-visible adjust track, border, and thumb styles.
  • ARIA: add role="switch" and keep aria-checked in sync when state changes via JavaScript.

Variants & Sizes

The system only exposes the base switch variant; dimensions are driven by the SCSS variables.

Base variant

  • No extra modifier classes; visual differences rely on .form-check-input and :checked.
  • States (checked, focus, disabled) are declared in the same SCSS file.

Sizing tokens

  • $form-sw-width 48px, $form-sw-height 24px, padding 3px.
  • $form-sw-thumb 16px, border-radius $Radius-100, transition duration $form-sw-dur.

Customization guardrails

  • Adjust dimensions only via the shared variables, not via inline styles.
  • New variants go through a DesignOps review before adding new SCSS selectors.

States

Only the documented states are implemented in SCSS.

Default

  • White track, border: 2px solid $Neutral-300, dark thumb, smooth transition.
  • Pressing applies transform: scale(0.92) via :active::before.

Checked / On

  • Border and track turn $Primary-500, thumb becomes white and scales (1.06).
  • Thumb shifts right using calc(100% - $form-sw-thumb - $form-sw-padding).

Hover

  • Adds box-shadow: 0 0 0 4px rgba($Primary-700, 0.12).
  • Colors adjust only within the defined palette.

Focus

  • Focus-visible applies box-shadow: 0 0 0 4px rgba($Primary-700, 0.22).
  • Focus owns the outline without overriding hover.

Disabled

  • Opacity 0.45, track $Neutral-100, border $Neutral-200.
  • Thumb turns $Neutral-400, box-shadow removed, checked disabled share $Neutral-200.

Styling documentation

Authoritative styling lives in css/common/_Switch.scss.

Selector scope

  • Targets .form-check.form-switch .form-check-input for the track, border, and relative positioning.
  • ::before renders the thumb.

Thumb & motion

  • Transitions use $form-sw-dur and $form-sw-ease for left/color changes.
  • Hover, focus-visible, and active states reuse the same transitions.
  • A media query removes transitions when prefers-reduced-motion: reduce.

Tokens

  • Colors rely on $Primary-500, $Primary-700, and $Neutral-*.
  • Sizing uses $form-sw-width, $form-sw-height, $form-sw-padding, $form-sw-thumb, and $Radius-100.

Labels

  • .form-check-label applies margin-inline-start: 10px.
  • Label spacing stays consistent in both layout directions.

Usage guidelines

Keep switches disclosed within the page layout rather than hiding them behind modals.

Placement

  • Place the label next to the switch so it reads as a single row.
  • Avoid stacking multiple switches without distinct labels.

Behavior

  • Apply changes immediately and show inline feedback for longer operations.
  • Announce significant side effects using aria-live regions when necessary.

Accessibility considerations

Guidance driven by the current markup.

  • Semantics: use input type="checkbox" role="switch" and keep aria-checked synchronized for scripted toggles.
  • Keyboard: rely on the browser to handle Space/Enter; do not intercept focus.
  • Focus visibility: the box-shadow in :focus-visible cannot be removed.
  • Labels: always provide a descriptive label that references the input id.
  • Disabled state: disabled and aria-disabled="true" share the same opacity and colors.

Do & Don’t

Do

  • Associate the label via for/id and keep the text concise.
  • Describe the current state when the effect is not obvious.
  • Preserve the provided focus shadow.

Don’t

  • Don’t repurpose switches as navigation controls.
  • Avoid invisible labels that rely solely on aria attributes.
  • Do not override the shared sizing tokens locally.

Edge cases

  • Switches that trigger asynchronous saves need a loading state or status message.
  • Programmatic toggles must keep aria-checked synchronized with the visual state.
  • In RTL, maintain the icon/label order; don’t flip them inside the control.
  • Honor prefers-reduced-motion: reduce to remove transitions when necessary.

Developer API

This section shows the required switch markup and ARIA attributes. 

<div class="form-check form-switch">
  <input class="form-check-input" type="checkbox" role="switch" id="autoSaveSwitch" aria-checked="false">
  <label class="form-check-label" for="autoSaveSwitch">Enable auto-save</label>
</div>
  • Use role="switch" plus synchronized aria-checked.
  • Keep the .form-check.form-switch wrapper for layout and spacing.
  • Do not remove focus-visible outlines or box-shadow.

Extension policy

  • New variants follow: proposal → review → implementation → documentation → approval.
  • Token adjustments only through css/common/_Switch.scss.
