System Navigation & Feedback

The platform-level navigation/feedback framework pairs the fixed Top Bar (Navbar), the Sidebar, and the Infra-sourced Primary Alert to deliver consistent identity, navigation, and critical messaging everywhere.

  • The Top Bar is a fixed, unified header that communicates system identity and delivers global search, layout controls, app switching, and user context across every experience.
  • System and project identity sit near the top-right (RTL) to clarify the active platform and project context.
  • A wide global search area in the center (placeholder “الاستعلامات العامة”) ties together queries over applications and the system’s core entities.
  • Layout controls, quick actions, notifications, and status icons live on the left edge so system-level actions stay within reach without leaving the current context.
  • The App Switcher, session context, and user profile close the Top Bar’s visual hierarchy and enable seamless cross-application navigation.
  • The Sidebar is the primary in-application navigation channel. It presents a hierarchical tree of modules/pages, supports nested sub-menus, reflects user permissions, and keeps the Top Bar focused by housing deeper navigation.
  • The Primary Alert comes from Infra, not the local UI. It surfaces critical system-wide messages—outages, maintenance, mandates—consistent across apps, governed by Infra, and must never be overridden at the app level.

Overview

The Navbar is a persistent application header that keeps identity, global search, layout controls, app switching, and user context within reach of every experience.

  • System identity and project context sit near the right edge (RTL).
  • A wide global search sits center stage with the placeholder “الاستعلامات العامة” for cross-application queries.
  • Layout utilities and quick actions gather on the left, alongside notifications and state icons.
  • The app switcher, session text, and profile complete the visual flow.

Anatomy

Each numbered block explains the region responsible for identity, search, actions, switching, and user context.

  1. 1. Navbar Container (الحاوية الرئيسية)

    Full-width, fixed-height white band with a subtle bottom shadow that hosts every top-level control and remains visible across the platform.

  2. 2. Brand & System Identity (أقصى اليمين – RTL)

    The rightmost edge renders the platform name, seal, and a subtitle that shifts depending on the active application to keep context obvious.

    2.1 Brand Block (الكتلة الخضراء)

    Displays the constant platform name "منصة أمن", the official logo, and a dynamic subtitle describing the current module (e.g., "إدارة المستخدمين والصلاحيات", "إدارة البلاغات والحوادث", "إدارة التصاريح والخدمات").

    • Platform name stays fixed while the subtitle updates with the active app.
    • The subtitle reflects the current functional scope and must never be treated as static text.
    • Delivers identity and clear context before the specific application shell loads.

    2.2 Sidebar Toggle

    Circular icon button beside the brand block toggles the sidebar expansion/collapse and thus adjusts the content width.

  3. 3. Global Search Area (منتصف الـNavbar)

    The large search input shows the placeholder “Search for people, vehicles, and weapons” alongside an icon, touching every people, vehicle, and weapon dataset. It is tuned for full focus, keyboard control, and pressing Enter immediately triggers the query.

    • Full-focus support plus Enter-to-search keeps the experience keyboard friendly.
    • Optional autocomplete or suggestion overlays for people, vehicles, and weapons appear without shifting the layout or breaking structure.

  4. 4. Utilities & System Actions (اليسار – RTL)

    The left zone collects quick-action icons, the app switcher, session context, notifications, and the user profile to close the visual journey.

    4.1 System Action Icons Group

    Icon-only buttons represent exit/back, fullscreen, settings, and notifications (with numeric badges). They rely on aria labels, consistent hit areas, and only appear when permission logic allows.

    4.2 App Switcher

    A grid-style button opens the app switcher dropdown showing permitted applications (icon, name, active state). Selecting an app updates the brand subtitle, loads the corresponding shell, and avoids full page reload unless required.

    4.3 Session / Context Info

    A read-only line such as "تسجيل أخر دخول/..." provides temporal session context right beside the actions.

    4.4 User Profile

    A circular avatar opens the user menu (Profile, Preferences, Logout) and serves as the contextual endpoint of the navbar.

Sidebar Structure

Every primary sidebar item is implemented as a single <li> that serves as the core structural container, grouping the visual label, icon, and any related behavior in a predictable block.

  • Inside that <li>, an icon-link wrapper defines the primary interactive area, housing the inline SVG icon, label, and optional expand/collapse arrow while managing hover, focus, and click interactions.
  • The <li> may optionally include a nested <ul> submenu for related secondary links; its presence is communicated through the expand/collapse arrow and the sidebar interaction logic.
  • This <li>-based structure is mandatory across the sidebar to keep alignment, interaction, accessibility, and scalability consistent system-wide.

All sidebar icons must be inline SVG, and every fill attribute (fill, fill="none", fill-opacity, etc.) must be removed; color is controlled through CSS inheritance (e.g., currentColor) so hover, active, disabled, and theme states stay predictable.

Governance note: any sidebar item or SVG icon that deviates from this structure or defines its own fill is non-compliant with the Design System and must be corrected before release.

DesignOps & Governance

The Sidebar structure is centrally managed by the Design System and serves as the single source of truth across applications; alternative structures or local variations are not permitted.

  • Every sidebar item must follow the approved pattern (single <li>, icon-link container for the interactive area, and optional nested sub-menu). No other structures are allowed.
  • Any change to structure or behavior must go through the DesignOps process and be reviewed by design and frontend to safeguard cross-application consistency and scalability.
  • Icon governance: use inline SVG only, remove every fill attribute (fill, fill="none", fill-opacity, hardcoded colors), and drive color exclusively through inherited CSS (e.g., currentColor).
  • This governance ensures consistent visual states, theming, accessibility, and predictable interaction behavior.

Compliance rule: any sidebar item or SVG icon that violates the approved structure or defines its own fill is non-compliant and must be corrected before release.

Structural consistency, accessibility, and scalability are mandatory—no application-level deviations are allowed.

Behavior Notes

  • Navbar is sticky and unaffected by page scrolling.
  • Application switching updates context without breaking the layout.
  • Elements appear/disappear based on permissions.
  • Fully keyboard accessible.
  • Icon-only buttons must not rely solely on tooltips.

Do

  • Keep icon sizes and spacing consistent.
  • Keep the search experience global and predictable.
  • Keep the brand block consistent across the platform.
  • Use the App Switcher for application-level navigation.

Don’t

  • Don’t overload the Navbar with actions.
  • Don’t hide critical actions without permission logic.
  • Don’t treat the subtitle as static text.
  • Don’t introduce alternative app-switching patterns.

Optional Layout Variants

  • Default layout with the expanded sidebar, showing icons and labels while keeping the navbar fixed.
  • Compact sidebar (icon rail) for tighter horizontal spaces.
  • Layout without a sidebar when standalone modules take over the full width.
