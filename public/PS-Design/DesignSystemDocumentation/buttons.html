Components / Actions

Buttons

Buttons are the primary control for user actions and interface interaction. This document outlines button usage guidelines, anatomy, states, and how to apply them effectively.

المكوّن: الأزرار الإصدار: قيد التحديث آخر تحديث: 28 يناير 2026

Overview

Buttons are essential interactive elements used to trigger actions. The button system is class-based, ensuring flexibility and consistency.

Base Classes

Use .moi-btn as the root class, then add exactly one .moi-btn--* modifier. The legacy .btn hook is still supported for backward compatibility, but it is not the canonical API in the current documentation.

Variants

Visual styles (such as color and appearance) are applied using a single modifier class starting with .moi-btn--*.

Button Variants & Usage

Button visual styles are defined by unique modifier classes. Choose the class that best suits the action context.

Core Variants

Variant CSS Class Usage
Primary .moi-btn--Primary For the most important and affirmative actions.
Neutral .moi-btn--Neutral For secondary or less emphasized actions.
Lavender .moi-btn--Lavender For alternate or secondary flows that need clear visual distinction.
Info .moi-btn--Info For informational or guidance-related actions.
Warning .moi-btn--Warning For actions requiring caution or warning the user.
Error .moi-btn--Error For destructive actions or indicating a critical error.

Solid Variants

The --solid suffix provides an alternate filled treatment where that modifier is defined in the SCSS.

Variant CSS Class Usage
Primary Solid .moi-btn--Primary--solid Alternate filled treatment for primary actions.
Neutral Solid .moi-btn--Neutral--solid Support and reset actions inside action groups.
Lavender Solid .moi-btn--Lavender--solid Secondary highlighted flows in the lavender palette.
Info Solid .moi-btn--Info--solid Visible informational actions without using the base info treatment.
Warning Solid .moi-btn--Warning--solid Warnings that need stronger visual emphasis.
Error Solid .moi-btn--Error--solid Destructive or corrective actions with strong emphasis.

Outline Variants

Outline button styles are available to reduce visual emphasis. Append --outline to the variant class.

Variant CSS Class Usage
Primary Outline .moi-btn--Primary--outline For secondary actions related to a primary action.
Neutral Outline .moi-btn--Neutral--outline For alternative actions or actions in a neutral context.
Warning Outline .moi-btn--Warning--outline To emphasize a warning while reducing visual clutter.
Info Outline .moi-btn--Info--outline To highlight information without the need for a solid button.
Error Outline .moi-btn--Error--outline For delete or undo actions requiring visual caution.

Developer API

This section details how to construct buttons using the available CSS classes.

Primary Button

<button class="moi-btn moi-btn--Primary">إرسال</button>

Neutral Outline Button

<button class="moi-btn moi-btn--Neutral--outline">إلغاء</button>

Button with Icon

<button class="moi-btn moi-btn--Primary">
  <span>المزيد</span>
  <svg width="20" height="20" viewBox="0 0 20 20" fill="none" aria-hidden="true">
    <path d="M8 5L13 10L8 15" stroke="currentColor" stroke-width="1.6" stroke-linecap="round" stroke-linejoin="round"/>
  </svg>
</button>

Icon-Only Button

<button class="moi-btn moi-btn--Neutral" data-icon-only="true" aria-label="Settings">
  <svg width="20" height="20" viewBox="0 0 20 20" fill="none" aria-hidden="true">
    <circle cx="10" cy="4" r="1.5" fill="currentColor"/>
    <circle cx="10" cy="10" r="1.5" fill="currentColor"/>
    <circle cx="10" cy="16" r="1.5" fill="currentColor"/>
  </svg>
</button>

Disabled Button

<button class="moi-btn moi-btn--Primary" disabled>معطل</button>
<button class="moi-btn moi-btn--Neutral" aria-disabled="true">معطل</button>

Anatomy

Structure is inferred from selectors in the SCSS and follows this basic structure:

Basic Structure

Every new button uses .moi-btn with exactly one .moi-btn--* modifier. 

<button class="moi-btn moi-btn--Primary">Button Label</button>

Icon Support

Buttons can contain icons (SVG or IMG) alongside text or as icon-only buttons. 

<!-- Button with text and icon -->
<button class="moi-btn moi-btn--Primary">
  Label <svg class="icon">...</svg>
</button>

<!-- Icon-only button (requires aria-label) -->
<button class="moi-btn moi-btn--Neutral" data-icon-only="true" aria-label="Settings">
  <svg class="icon">...</svg>
</button>

Base Styles

The current canonical root is .moi-btn, which calls @include buttonStyles; in _buttons.scss. The system still supports the legacy path .btn[class*="moi-btn--"] for backward compatibility only.

Sizing

Button size is controlled via the data-size="sm|md|lg|xl" attribute. When omitted, the button defaults to md size.

Size Examples

<button class="moi-btn moi-btn--Primary" data-size="sm">Small Button</button>
<button class="moi-btn moi-btn--Primary" data-size="md">Medium Button</button>
<button class="moi-btn moi-btn--Primary" data-size="lg">Large Button</button>
<button class="moi-btn moi-btn--Primary" data-size="xl">Extra Large Button</button>

Touch Target

A pseudo-element (::after) enforces a minimum 44x44 pixel touch target, improving accessibility on touchscreens without altering the button's visual size.

Icon-Only Buttons

To enable an icon-only button, use the data-icon-only="true" attribute and provide a descriptive aria-label. 

<button class="moi-btn moi-btn--Neutral" data-icon-only="true" aria-label="Settings">
  <svg class="icon">...</svg>
</button>

Disabled Behavior

The disabled styling is applied on .moi-btn:disabled and also covers aria-disabled="true": background $Neutral-100, border 1px solid $Neutral-300, and text color $Neutral-400.

Accessibility

Guidance based on the current CSS implementation.

  • Use <button> for interactive elements that perform an action. Only use <a> for navigation to other destinations.
  • Icon-only buttons must be provided with a descriptive aria-label.
  • Maintaining visible focus outlines is crucial for keyboard users. Avoid removing default :focus styles.
  • For disabling a native button, use the disabled attribute. For non-button elements that are visually disabled, use aria-disabled="true".
  • Do not rely on color alone to convey meaning or state. Use supplementary text or icons.
  • Keep focus outlines for variants that define them: .moi-btn--Neutral, .moi-btn--Neutral--outline, and the capitalized palettes (.moi-btn--Primary*, .moi-btn--Lavender*, .moi-btn--Warning*, .moi-btn--Info*, .moi-btn--Error*).

Usage Guidelines

Solid vs outline

  • Use solid variants (.moi-btn--Primary, .moi-btn--Neutral) for primary actions.
  • Use outline variants for secondary or adjacent actions.

Palette coverage

  • The currently approved palettes are Neutral, Primary, Lavender, Info, Warning, and Error only, with their defined --solid and --outline modifiers where available.

Status actions

  • Use danger and warning variants for destructive or cautionary actions.
  • Use info variants for informational actions.

Theming

  • Variants using var(--theme-primary) inherit theme color.

DesignOps Governance

Governance framework ensuring consistency, quality, scalability, and accessibility for buttons.

DesignOps Governance

Framework to ensure a unified, scalable experience across products.

Quick Checks

  • Every button = .moi-btn + exactly one modifier class.
  • Sizes are controlled solely via the data-size attribute.
  • Touch target must meet a minimum of 44x44 pixels.
  • Supported states: Default, Hover, Active, Focus-visible, Disabled.
  • No changes or additions without DesignOps review and official documentation.
  • For full governance details, refer to GOVERNANCE.md.

Purpose

  • Ensure a unified, scalable experience across products.
  • Maintain visual and functional quality.
  • Apply accessibility requirements systematically.

Scope

Applies to all buttons using .moi-btn + .moi-btn--* across all variants, states, and platforms.

Ownership & Responsibilities

  • Design Owner: Visual decisions, accessibility, variants, and styles.
  • Engineering Owner: Implementation, performance, compatibility.
  • DesignOps: Process, approvals, documentation.

Usage Rules

Each button = .moi-btn + exactly one modifier (.moi-btn--*).

Allowed

<button class="moi-btn moi-btn--Primary">حفظ</button>

Disallowed

<button class="moi-btn moi-btn--Primary moi-btn--Warning">حفظ</button>

Naming Governance

  • Naming is semantic, not contextual.
  • No feature-based or visual-description class names.
  • Modifiers are the only way to express variants and styles.

Sizing & Spacing Governance

  • Size is controlled only via data-size="sm|md|lg|xl".
  • No local padding or height overrides.
  • Spacing, heights, and icon sizing are governed by design tokens.
  • Touch target must meet 44×44 via hit-area expansion without changing visual size.

State Governance

  • Supported states: Default, Hover, Active, Focus-visible, Disabled.
  • Hover must never replace focus-visible.
  • Disabled and aria-disabled share the same visual treatment.
  • No custom or undocumented states.

Accessibility Governance

  • Focus visibility is mandatory.
  • Icon-only buttons require an aria-label.
  • Use <button> for actions and <a> for navigation.
  • Color alone must not convey meaning.

Extension Policy

  • New variants or behaviors are allowed when clearly justified.
  • Requires formal flow: proposal → design review → token mapping → implementation → documentation → approval.
  • No direct additions without approval.

Documentation Rules

  • Any change must be documented in the same PR.
  • Code examples are authoritative.
  • Undocumented classes are not allowed.
  • Documentation avoids historical or transitional language.

Compliance

  • Violations are treated as Design Debt.
  • DesignOps may request changes or block merges.

TL;DR Summary

  • One button = .moi-btn + one modifier.
  • Sizes via data-size only; touch target 44×44.
  • States limited to five; focus must be visible.
  • Icon-only requires aria-label.
  • No changes or additions without review and documentation.

Do & Don’t

Do

  • Use .moi-btn with one approved modifier only.
  • Preserve focus outlines where they exist.
  • Use icon filters and svg fills to maintain contrast.

Don’t

  • Use text buttons for primary actions.
  • Remove focus outlines or override them with custom styles.
  • Stack multiple modifiers on a single button.
