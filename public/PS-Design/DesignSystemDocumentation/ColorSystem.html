Color System

The color system provides a stable reference for surfaces, text, and states. It relies on SCSS tokens to keep UI consistent and scalable across components.

Token Mapping

Palette swatches map directly to the SCSS tokens defined in css/helpers/_variables.scss. Each table row includes the variable name, the HEX value, and utility classes (`.bg-*`, `.border-*`, `.text-*`) so developers can copy the right token without guessing.

  • Token names like $Primary-500 are applied through helpers such as .bg-Primary-500 and .border-Primary-500.
  • Semantic tokens (Info, Warning, Error) convey states regardless of direction.
  • Stay within the palette and preserve contrast instead of inventing new colors.

Overview

The color system provides a scalable, consistent palette for UI surfaces and content. Colors are defined as SCSS tokens (for example, $Primary-500) so the tokens remain the single source of truth in design and implementation.

Color is a medium for expressing style, evoking emotion, and conveying significance. Intentional, standardized color usage creates a recognizable and pleasant experience.

Use darker shades for text and emphasis, lighter shades for backgrounds and states. Maintain accessible contrast, and prefer semantic usage (Primary for actions, Info for guidance, Warning/Error for critical feedback).

Color should always serve a clear purpose that supports a message or status a user can quickly identify.

A neutral foundation for surfaces creates a clean canvas so colored elements stand out and draw attention.

Color Roles

Translate colors into clear roles across every interface to make decisions faster and keep a consistent visual rhythm.

Surface

Surfaces

Use light neutrals for primary backgrounds to keep content legible.

Border

Borders

Use mid-tone borders to separate cards and interactive regions.

Text

Text

Use darker shades for body text and stronger tones for headings.

Anatomy

  • Token name: prefixed by category (e.g., $Neutral-, $Primary-).
  • Scale: numeric steps like 25-950 for color intensity.

Palettes

Colors are organized into neutral, primary, and semantic palettes to clarify intent and support consistent usage across products.

  • Neutral: base surfaces, text, and layout structure.
  • Primary: brand identity and key actions.
  • Semantic: states such as info, warning, error, and success.

Semantic Colors

Semantic colors communicate status quickly: red for errors, yellow for warnings, blue for information, and green for success. Keep usage consistent and pair with supporting text or icons.

Info

Guidance and clarification with light backgrounds and clear text.

Warning

Attention needed without overwhelming the primary message.

Error

Critical errors that require fast resolution with strong contrast.

Success

Success states use the primary palette to reinforce positive outcomes.

Visual

Color palettes rendered as swatches.

Use tonal scales from light to dark to express hierarchy across surfaces, borders, and text. Favor lighter tints for large backgrounds, mid tones for components, and darker shades for emphasis and content. Keep contrast in mind to ensure readability and consistent UI rhythm.

Contrast & Background Governance

Governance must embed enforceable contrast and background rules for every interface.

Governance Scope

  • Rules apply to all pages, components, and UI states.
  • Governance rules override visual preference or subjective design decisions.
  • The Contrast Checker is a validation tool, not the source of truth.

Minimum Contrast Thresholds (WCAG)

Normal text
4.5:1
Below 18px or below 14px bold
Large text
3:1
18px+ or 14px+ bold
UI essentials
3:1
Icons, indicators, borders, essential components
Attention:

Any result below the thresholds is a release-blocking failure. Exceptions require formal approval + documented accessibility review.

Background & Color Governance Rules

These requirements establish the minimum foundation for background and text contrast.

🎯

Do

Essential
  • Use simple, neutral backgrounds that support clarity and reduce noise.
  • Neutral / surface colors must be the default choice for page and section backgrounds.
  • Backgrounds must serve usability and function, not decoration.

Don’t

Forbidden
  • Primary or highly saturated colors must not be used as backgrounds for large areas.
  • Long-form text is not allowed on colored or visually strong backgrounds.
  • Decorative, textured, or visually noisy backgrounds are not permitted.
  • Backgrounds must serve usability and function, not visual decoration.

Color Selection & UI Behavior Rules

Color selectors must follow these interaction rules.

🎨
  • When selecting colors, always show visual color swatches next to each color.
  • Colors must be displayed as real visual samples, not code values.
  • When opening a color selector, display all colors as fixed div-based swatches, not dropdown options.
  • Do not use native <option> elements for color selection.
  • Display colors in a static swatch grid (similar to contrast swatches).
  • Selected colors must have a clear visual indicator (border, checkmark, or focus state).

DesignOps Governance Enforcement

These enforcement rules are non-negotiable and require documentation for any divergence.

🛡️

These rules are a release gate. Any failure = blocked release + documented evidence.

  • Contrast compliance is mandatory for all UI elements.
  • Any color or token that fails contrast checks is not allowed in production.
  • Contrast validation is required during: Design review; QA review; Pre-release approval.
  • Tokens failing contrast rules must be flagged, restricted, or documented.
  • Governance rules must live in the Design System, not inside tools.

QA Acceptance Criteria

Every release must prove compliance with the following measurable criteria.

  • All normal text meets a minimum contrast ratio of 4.5:1.
  • Large text meets a minimum contrast ratio of 3:1.
  • Icons, borders, indicators, and essential UI elements meet 3:1 contrast.
  • No screens use primary or saturated colors as large background areas.
  • Long-form text is never placed on colored or visually strong backgrounds.
  • Decorative or textured backgrounds are not present in any screen.
  • All contrast checks are verified using the Contrast Checker.
  • Any contrast failure results in a blocked QA status.
  • Screens cannot be approved or released unless all criteria pass.
  • Contrast validation evidence is documented as part of QA results.

Usage rules

Colors

  • Use $Primary-* for primary actions.
  • Use $Error-* and $Warning-* for status.
  • Tokens are defined inside css/helpers/_variables.scss; reference the name rather than raw HEX.

Do & Don't

Do

  • Use the defined scale tokens for consistency.
  • Apply palette steps in increasing intensity order.
  • Centralize new values in the variables file.

Don't

  • Hardcode hex values in components.
  • Create ad-hoc spacing values outside the scale.
  • Mix palette steps randomly within a single component.

Accessibility

Contrast between text and background is critical for readability. Choose tones that meet WCAG requirements.

  • Use sufficient contrast when pairing text with background tokens.
  • Prefer mid-range palette values for text and stronger values for accents.
  • Avoid using low-contrast combinations for critical status indicators.
  • Small text: below 24px must reach at least 4.5:1 contrast.
  • Large text: 24px and above should be 3:1 or higher.
  • UI elements: buttons, borders, and icons need at least 3:1.

RTL notes

Direction does not alter the palette: tokens such as $Primary-500 function the same in RTL and LTR interfaces.

  • Flip directional gradients or shadows only when the UI dictates; color intensity remains constant.
  • Keep the same token names across languages so translation notes stay aligned.
