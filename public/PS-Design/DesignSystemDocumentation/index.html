AMN Design System

Introduction

The AMN Design System is the official, SCSS-driven source of truth for all government security products under the AMN umbrella. It unifies visual language, interaction patterns, and accessibility standards for security officers, investigators, operational teams, and partner entities.

Platform Scope and Coverage

AMN is a multi-system platform; this system governs every UI delivered within it.

  • The Design System ensures consistency across divergent operational contexts—administrative, investigative, field, and shared services.
  • All AMN products must consume the same token and component library to prevent drift.

Design Philosophy

  • Trust and authority: interfaces must communicate institutional reliability for high-stakes decisions.
  • Clarity under pressure: layouts favor scannability, explicit labels, and predictable controls.
  • Error prevention: defensive defaults, confirmation for destructive actions, and clear state messaging.
  • Predictability: identical behaviors across all AMN systems to reduce training and operational risk.

Single Source of Truth (SCSS-First)

  • Tokens (color, spacing, typography, radius, shadows, motion) are defined and versioned in SCSS.
  • Variables and mixins enforce consistent application and safe overrides for RTL, density, and states.
  • Propagation: updates to SCSS tokens cascade safely through compiled CSS across all dependent systems.

Design System Architecture

Tokens

Defines the core visual primitives and motion values used across AMN surfaces.

  • Tokens: colors, typography scale, spacing, radius, shadows, motion curves/durations.

Foundations

Establishes layout, type, and accessibility rules for a unified experience.

  • Foundations: grid and layout rules, typography hierarchy, accessibility requirements (WCAG-aligned).

Components

Provides reusable controls and patterns for operational workflows.

  • Components: buttons, form controls, alerts, tables, modals, status indicators, system-state banners.

Utilities

Delivers helper classes and mixins for spacing, states, responsiveness, and RTL.

  • Utilities: helpers for spacing, typography, states, responsiveness, and RTL.
Implementation

SCSS structure: dedicated folders for tokens, mixins, utilities, and component partials compiled into main.css.

Design ↔ Development Workflow

  • Figma-to-SCSS parity: every token in design files maps to SCSS variables; new tokens require design and code approval.
  • Versioning: semantic releases with change logs per system; consuming apps pin versions and upgrade through controlled windows.
  • Breaking changes: introduced only with deprecation periods and migration notes; enforced by pull-request checks.

UI/UX Standards for AMN

  • Accessibility: WCAG-aligned contrast, focus visibility, keyboard support, and clear error messaging.
  • Responsiveness: layouts adapt to operational screens and tablets; dense data views maintain readability.
  • States: consistent patterns for success, warning, error, and critical states across all components.

Governance and Ownership

  • Ownership: Design System team owns tokens, components, and release cadence.
  • Reviews: mandatory design and code review for any token or component change.
  • Contributions: system teams submit proposals with usage evidence; approvals required before merge.
  • Boundaries: custom overrides outside the SCSS layer are prohibited unless formally approved.

Usage Guidelines

  • Mandatory adoption: all AMN systems consume the shared tokens and components; divergence is not permitted.
  • Custom solutions: only allowed when approved and back-ported into the Design System for reuse.
  • Requests: new tokens/components follow a formal request → review → implementation → release path.
