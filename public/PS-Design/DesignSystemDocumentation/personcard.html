Person Card

Core identity card that presents person data consistently and uses official SCSS tokens for colors and states.

المكوّن: بطاقة الشخص الإصدار: قيد التحديث آخر تحديث: 28 يناير 2026

Overview

The Personal Profile Card supports verification, inquiry, case management, inspection, and action scenarios.

Component Objective

  • Provide fast access to critical data, surface person status instantly, and enable confident decisions.
  • Reduce input/interpretation errors and keep presentation consistent across systems.

Use Cases

  • Approvals and Requests
  • Person Records
  • Visas and Residencies
  • Security Screening
  • Incident and Report Processing
  • Travel and Movement Review

Component Structure

Header

  • Status badge (Required/Not Required) is always visible.
  • 1:1 image, full name, and management actions gated by permissions.

Action Buttons

These are not tabs; each button opens an inline panel and multiple panels may remain open.

  • Person History
  • Residency Status
  • Visa Type
  • Travel Status
  • Complete Missing Data

Buttons hide when API data or permission is missing. Panels never render empty and reload only on status change.

Personal Information

Fixed order, no placeholders, missing fields hidden, and person status always visible.

Identity Information Block

Includes issue number, issue/expiry dates, issuing authority, and validity status (YYYY-MM-DD).

Design Guidelines

  • 8px grid, 1.5px icon stroke, and official tokens.
  • Neutral-50 backgrounds, Text-700 typography, Primary highlights.
  • Image size 96×96 with AA/AAA contrast.

Design Anatomy

Structure derived from Person Card SCSS while keeping the 8px grid.

  • Root: .person__card with 24px padding and vertical stacking on the 8px grid.
  • Header: 1:1 image, name, and status; actions row is optional and can be removed in Add/Manage scenarios.
  • Chips/quick fields: .person-card__chips surface status indicators or quick data.
  • Data grid: .person__card__Details uses responsive 12/6/3 columns with show/hide for extended details.
  • Extended identity block: identity details appear on reveal, without embedding business requirements inside.
Visual tokens
  • Outer radius 18px; inner tiles and chips use rounded 10–12px corners.
  • Header/content padding follow padding-XL / padding-L utilities (8px grid).
  • Spacing is 8–12px between elements; detail grid relies on bootstrap g-3.
  • Typography uses official Primary/Neutral colors; states use regpriority/highpriority badges.

DesignOps Governance – Person Card Usage (Inquiry, Add, Manage)

The Person Card is a governed Design System component whose behavior varies by scenario to support usability and operational scale; any deviation requires DesignOps approval.

Overview

  • The Person Card is a core and foundational system component.
  • Its behavior varies by scenario to support usability and operational scale.
  • Any deviation requires DesignOps approval.

Scenario-Based Rules

Apply the correct mode to keep core identity data consistent and auditable.

Inquiry / View

  • The Person Card is read-only.
  • MUST display ALL available person details.
  • Optimized for inquiry, investigation, and review use cases.

Add / Create & Manage

  • The Person Card MUST display ONLY core person information.
  • Default core fields (reference only):
    • National ID
    • Age
    • Gender
    • Nationality

Business Requirements Placement

  • Business requirements, validations, conditions, attachments, or feature-specific inputs MUST NOT be placed inside the Person Card.
  • They MUST be placed UNDERNEATH the card in a separate section.
  • The section MUST include a clear descriptive title (e.g. “Business Requirements”, “Additional Requirements”, “Feature Details”).
  • Visual separation must be clear but must not alter the card design.

Rationale (Why This Rule Exists)

  • Maintains a clean and scannable identity surface.
  • Reduces cognitive load during data entry.
  • Allows business logic to evolve without breaking the core component.
  • Enables DesignOps to audit and govern component usage at scale.

Acceptance Criteria (DesignOps Validation Checklist)

Use this checklist to validate every Person Card implementation by scenario.

Inquiry / View

  • Card displays ALL available person details.
  • Card is read-only.
  • No business logic embedded inside the card.

Add / Create & Manage

  • Card displays ONLY core information (max 4 fields).
  • No business requirements inside the card.
  • Business requirements appear BELOW the card.
  • A visible descriptive section title exists.
  • Core data and business inputs are clearly distinguishable.
  • Adding/removing requirements does NOT affect the card structure.
  • Rules are consistently applied across all products/features.
  • You can remove the person card actions row when needed.
