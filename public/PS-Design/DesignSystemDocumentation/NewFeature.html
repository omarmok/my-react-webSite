Layouts / Features

New Feature

Governed reference for Service Page structure, defining mandatory rules for search, tabs, table layout, and loading states.

المكوّن: ميزة جديدة الإصدار: قيد التحديث آخر تحديث: 28 يناير 2026

Overview

A Service Page displays searchable service data in a unified layout. Consistency lowers cognitive load, improves usability, and enables DesignOps auditing.

  • Layout is standardized and governed by DesignOps; exceptions require approval.
  • Defines what users see: title, search, optional tabs, header bar, table, pagination.
  • Acts as a single contract for designers, engineers, and auditors.

Design Anatomy

Required top-to-bottom order:

  1. Page Title
  2. Search Block
  3. Tabs (optional)
  4. Table Header Bar
  5. Table
  6. Pagination

Search Block Rules

  • Max fields: 4 (preferred 3).
  • Required buttons: Search (primary), Advanced Search (secondary), Reset (tertiary).
  • Extra filters live inside Advanced Search only.
  • Mirror the reference row: labeled inputs with action cluster on the right.

Tabs Rules

  • Optional; only for status/category switching.
  • Use the approved Tabs component with standard spacing.
  • Keep labels concise; no inline actions inside tabs.

Table Anatomy

Table Header Bar

  • Right: total records count.
  • Left: Customize, Print, Export (optional).
  • Keep alignment and standard spacing.

Body & Columns

  • Column order is consistent across pages.
  • Status badges follow design system tokens.
  • Use a dedicated actions column if needed.

Column Limit Rule

  • If columns exceed 8, provide a column visibility selector.
  • Save column preferences per user.
  • Default view prioritizes the most critical columns first.

Loading State

Add the class tableloader while data loads; remove it when ready.

  <!-- rows here -->
</table>

Empty State

  • Keep table structure visible.
  • Show “No Data” per design system pattern.
  • Record count must show 0.

Governance Reminder

Do not hide the header bar or pagination when empty.

DesignOps Governance

  • Ownership: Design System / DesignOps teams manage layout and reviews.
  • Change management: any change requires approval and documentation.
  • Prohibited deviations without prior approval.
  • Track design debt with an agreed remediation date.
  • Compliance and periodic auditing are the responsibility of delivery teams.

Do & Don’t

Do

  • Follow the mandatory anatomy order.
  • Keep 3–4 search fields with required buttons.
  • Use the header bar and consistent columns.
  • Enable column selector when >8 columns.
  • Apply tableloader during loading.

Don’t

  • Add extra search fields to the main row.
  • Reorder columns arbitrarily.
  • Invent custom loading behavior outside tableloader.
  • Hide structure or pagination on empty states.
  • Ship deviations without DesignOps approval.
